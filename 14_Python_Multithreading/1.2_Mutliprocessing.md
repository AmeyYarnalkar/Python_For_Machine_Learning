
Multiprocessing simply means:

 **Run multiple Python processes instead of multiple threads.**

Each process has:

* its own Python interpreter
* its own memory
* its own CPU core

So CPU-heavy tasks actually run **in parallel**.

---

## 1. What is Multiprocessing

Normal program:

```python
task1()
task2()
task3()
```

Execution:

```
task1 → finish
task2 → finish
task3 → finish
```

Multiprocessing:

```
task1  \
task2   → run on different CPU cores
task3  /
```

If your computer has **4 cores**, Python can run **4 processes simultaneously**.

---

## 2. Basic Example

Python module:

```python
import multiprocessing
```

Simple program:

```python
import multiprocessing

def task():
    print("Process running")

p = multiprocessing.Process(target=task)

p.start()
p.join()
```

Explanation:

* `Process()` → create process
* `start()` → start execution
* `join()` → wait until process finishes

---

## 3. Multiple Processes

```python
import multiprocessing

def work(n):
    print("Working on", n)

p1 = multiprocessing.Process(target=work, args=(1,))
p2 = multiprocessing.Process(target=work, args=(2,))

p1.start()
p2.start()

p1.join()
p2.join()
```

Now the tasks run **in parallel**.

---

## 4. Real Example (CPU Work)

Example: square numbers.

```python
import multiprocessing

def square(n):
    print(n*n)

processes = []

for i in range(5):
    p = multiprocessing.Process(target=square, args=(i,))
    processes.append(p)
    p.start()

for p in processes:
    p.join()
```

Each number can run on a **different core**.

---

## 5. Process Pool (Most Common Way)

Instead of manually managing processes, use a **Pool**.

```python
from multiprocessing import Pool

def square(n):
    return n*n

with Pool(4) as p:
    result = p.map(square, [1,2,3,4,5])

print(result)
```

Output:

```
[1, 4, 9, 16, 25]
```

Explanation:

* `Pool(4)` → create 4 worker processes
* `map()` → distribute tasks automatically

This is the **most practical multiprocessing pattern**.

---

## 6. Why Multiprocessing is Faster for CPU Work

Example task:

```
compute large prime numbers
matrix multiplication
video encoding
image processing
```

Threads:

```
1 CPU core used
```

Processes:

```
multiple CPU cores used
```

So multiprocessing **bypasses the GIL**.

---

## 7. Important Rule (Very Important)

In Python multiprocessing you must use:

```python
if __name__ == "__main__":
```

Example:

```python
import multiprocessing

def task():
    print("Hello")

if __name__ == "__main__":
    p = multiprocessing.Process(target=task)
    p.start()
    p.join()
```

Without this, programs can **crash or create infinite processes**, especially on Windows.

---

## 8. Multiprocessing vs Multithreading

| Feature    | Multithreading | Multiprocessing |
| ---------- | -------------- | --------------- |
| Memory     | Shared         | Separate        |
| Speed      | Good for I/O   | Good for CPU    |
| GIL impact | Yes            | No              |
| Overhead   | Low            | Higher          |

Rule of thumb:

```
Network / file / API → threads
Heavy computation → processes
```

## 9. Real Example Use Cases

Multiprocessing is used for:

* Data processing pipelines
* Machine learning preprocessing
* Image/video processing
* Scientific computing
* Simulation tasks

Libraries like **NumPy, Pandas, PyTorch** internally use multiprocessing or C-level parallelism.

### Why Multiprocessing is Not Ideal in Jupyter Notebook

1. **Multiprocessing expects a script file**
   It is designed to run programs like `python script.py`.
   Notebooks run code in cells, not in a real `.py` file.

2. **Child processes try to import the main program**
   When a new process starts, Python tries to re-import the **main module**.
   In notebooks there is no proper module file to import.

3. **The environment is dynamic**
   Notebook code runs in cells that execute one by one.
   Child processes cannot reliably recreate that environment.

4. **`__main__` guard becomes tricky**
   Multiprocessing normally needs:

   ```python
   if __name__ == "__main__":
   ```

   In notebooks the execution model is different, so this guard
   does not behave as cleanly as it does in scripts.

5. **Print output may not appear**
   Child processes have their own stdout.
   So `print()` inside a process might not show in the notebook cell.

6. **Kernel state causes inconsistencies**
   The notebook kernel keeps variables and functions in memory.
   Spawned processes may not see the same state.

7. **Debugging becomes harder**
   Errors inside child processes often do not show clearly in notebooks.

---

### Common Practice

Developers usually follow this workflow:

```
Jupyter Notebook → experiment and prototype
Python script (.py) → run multiprocessing code
```

Multiprocessing works **much more reliably in normal Python scripts**.
